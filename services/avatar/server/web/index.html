    How cost prediction is calculated

    $/sess-hr is an estimate of GPU-only cost to run one concurrent talking session for one hour, derived from the live bench on this machine — not a cloud invoice.

    1. Measure busy_ratio during the 10-dialogue (or Capture) run. Continuous models (FlashHead): chunk_gen_ms ÷ chunk_duration. Clip models (Ditto / MuseTalk / Wav2Lip): utterance_render_ms ÷ audio_duration. Example: 6 s of GPU work for 2 s of speech → busy_ratio ≈ 3.0.
    2. Convert to effective sessions per GPU with 15% headroom: eff_sessions = 0.85 ÷ busy_ratio, also capped by VRAM (0.9 × VRAM_total ÷ VRAM_used). This value is fractional — below 1 means the model is slower than realtime (you need more than one GPU-hour per session-hour of speech).
    3. GPU $/session-hour = $0.44 ÷ eff_sessions (RunPod-class RTX 4090 Secure rate used in this project). So FlashHead at busy≈0.48 → ~1.77 sessions → ~$0.25; a slow clip model at busy≈2.7 → ~0.31 sessions → ~$1.42.
    4. Scale projection (right panel) layers non-GPU stack costs on top for fleet planning: LiveKit minutes, TTS chars, LLM turns — using the sliders for concurrency / talk fraction. Matrix $/sess-hr stays GPU-only so models compare fairly.
    Inputs from testing
    • gen_ms / utterance render times

    What it is / isn’t
    • Is: relative GPU cost under our harness
    • Is: sensitive to real measured load
    • Isn’t: TTS/LLM/network invoice
    • Isn’t: TensorRT / multi-tenant packing magic
